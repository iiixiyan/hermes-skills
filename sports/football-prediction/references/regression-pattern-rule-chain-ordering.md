# 规则链顺序回归模式（2026-07-07发现）

> 当在引擎中插入新规则时，现有规则检索执行顺序被改变导致的回归问题。

## 问题背景

v10.35引擎新增了F38/F40/F41/F42四条规则，插入到`_predict_raw()`函数的多个位置。这些新规则的插入改变了现有if/elif链的执行路径，导致3个此前正确的历史比赛被错误拦截。

## 模式归因

### 模式A — P11-Hot-B `pass` → R6-BL拦截（⚠️ 最常见）

**触发链路：**
```
P11-Hot-B块(line 700): 条件 xa+c1≥25+fh≤15+fd≥15+o1<5.0
  → bl+hd+r2h+not is_r1 → pass (造热完满,让后续R6-BL-Trap处理)
  → 跳出P11-Hot-B块 → 执行line 726起后续代码
  → 到达R6-BL块(line 948): bl=True → 进入
  → fh≤5分支: 假设精英队碾压 → 返回(3,1) ❌ 实际1-2
```

**根因：** `pass`的注释说"让后续R6-BL-Trap处理",但R6-BL的fh≤5分支并未检查淘汰赛三极端诱盘条件。

**修复方法：** 在R6-BL块的`fh≤5`分支前插入R16-KO-TRAP检查:
```python
if bl:
    if rd == 0 and hd and xa and r2_high and 10 <= fd <= 40 and fa > 20:
        return (1, 2, "R16-KO-TRAP", 4)
    if fh <= 5:
        ...
```

### 模式B — 文档化规则未在引擎实现

**案例1：因子36(c3=0)**
- SKILL.md有完整文档: `c3=0+hd≥45+xa≥45→极致造热反转,客胜概率>平局`
- 但R6-BL-Super-Consensus-H分支仅检查`c1≥43`就返回(3,1),未检查c3=0条件
- 修复: 在`if c1>=43:`分支内加c3==0检查

**案例2：因子37(客胜极端升幅)**
- SKILL.md有完整文档: `客胜升幅>35%+c1≥40+主即赔<1.40→主队λ_h+1`
- 但R6-BL-HD-Boom分支返回(5,1)未做进球上限制约
- 修复: c3<5+c1≥40+o1<1.40时返回(3,0)

### 模式C — 新规则插入位置影响下游

新增规则插入if/elif链中段时,会改变后续所有elif的匹配概率。F38插入在R15前,R16前,Default前——这些块的条件可能因为F38提前return而永不执行。

## 预防清单（每次新增规则后执行）

```
□ 新增规则插入在if/elif链的什么位置？
   - 链头(第1个elif)? → 影响全链,必须全量回归
   - 链中? → 检查下游规则是否被"屏蔽"
   - 链尾(Default前)? → 影响最小,但可能被链中其他规则拦截
□ 新增规则是否和其他已有规则共享输入条件？
   - 若有条件重叠 → 当前规则排前面则拦截,后面则永不触发
   - 重叠条件+不同输出 → 必须决定优先级
□ 新增规则所在块是否有`pass`逻辑让执行流跳转到其他块？
   - 检查pass后的所有后续块是否理解这个`pass`的含义
   - 如果pass表示"让后面处理"→ 后面块必须确实能处理
□ 全量回归是否覆盖了"前版本正确的所有已知案例"？
   - 不能只验证新增场次,必须回放全部历史
   - 建议保留固定回归测试集
```
