# SkillOpt 训练循环 — 竞足预测技能自适应优化

> 基于微软研究院 SkillOpt 论文的工程化适配：把复盘→优化视为文本空间里的训练算法。
> 论文：https://arxiv.org/abs/2605.23904

## 核心映射

| SkillOpt 概念 | 竞足场景实现 | 管控机制 |
|:-------------|:------------|:---------|
| **Rollout batch** | 当日预测执行（~9场比赛/批） | cron定时执行 |
| **Trajectory logging** | 预测→实际比分→偏差归因 | `review-findings-YYYYMMDD.md` |
| **Minibatch reflection** | 按联赛分组复盘（≥3场分组分析） | 联赛模板强制 |
| **Text learning rate Lₜ** | 每轮最多 **3 条**编辑进入 skill | 编辑预算计数器 |
| **Validation gate** | 次日验证：旧skill vs 新skill 在同一数据集回测 | 必须严格涨分 |
| **Rejected buffer** | 被拒绝的编辑方案 → `10-rejected-edits-buffer.md` | 避免重复踩坑 |
| **Slow/meta update** | 周/月复盘 → 跨联赛规则迁移 → skill 版本号提升 | 周复盘强制 |
| **best_skill.md** | SKILL.md 当前稳定版本（v4.x.y） | 版本号控制 |

## 一、训练循环流程

### Step 1: Rollout — 执行预测（每日 cron）

```
cron cd0456f2c803 (20:00 预测) → 输出预测结果
```

**记录内容**（每场至少保留）：
- 预测比分、胜平负、让球
- 触发因子清单（4b/12/13/14/15/16/17…）
- 数据时效标记
- 信心评级

### Step 2: 轨迹采集 — 获取实际结果（赛后 cron）

```
cron f84b9cf8d1ef (10:00 复盘)
```

**采集**：59itou 开奖结果页 + 重载确认所有场次出齐。

**轨迹格式**（每场一条）：
```json
{
  "match_id": "5001",
  "league": "瑞超",
  "predicted": {"score": "1-1/2-1", "direction": "胜", "handicap": "让平"},
  "actual": {"score": "2-2", "direction": "平", "handicap": "让胜"},
  "factors_triggered": ["4b升客胜", "12降主胜"],
  "deviation": "差1球",
  "deviation_cause": "λ客被高估0.5球"
}
```

### Step 3: Minibatch Reflection — 按联赛分组找共性偏差

**规则**：
1. 按联赛分组（挪超一组、瑞超一组、国际赛一组…）
2. 每组≥3场 → 完整模板；<3场 → 简要模板
3. 每组独立做偏差矩阵分析

**必查的共性偏差模式**：
- 某联赛全部被某一因子误导 → **因子失效条件探索**
- 某联赛的某个λ系数系统性偏差 → **λ校正系数调整**
- 某联赛的平局率/主胜率偏离模型预期 → **联赛基因参数修正**

### Step 4: 编辑提案（最多 3 条）

从偏差矩阵中筛选可迁移的程序性规则。每轮最多输出 **3 条编辑提案**。

每一条提案格式：
```
编辑提案 N/N：
  目标：references/xx.md §Y 或 SKILL.md §Z
  操作：add | delete | replace | patch
  内容：[具体文本修改]
  预期收益：[量化，如「预计精确命中率提升X%」]
  验证方式：[如「回测过去5场同一条件的所有比赛」]
```

### Step 5: 验证集门控

**门控流程**：
1. 用当前 skill（旧版）在验证集上跑一遍 → 得分 S_old
2. 应用编辑提案，生成候选 skill（新版）
3. 用候选 skill 在验证集上跑一遍 → 得分 S_new
4. 只有当 **S_new > S_old**（严格大于，打平也算不通过）时，才接受编辑

**验证集定义**：
- 日常复盘：当天所有比赛的偏差回归数据作为验证集
- 周度复盘：过去 N 天有明确偏差记录的比赛作为验证集
- 月度复盘：全月轨迹数据作为验证集

**门控通过** → 正式写入 SKILL.md/references
**门控拒绝** → 移入 rejected buffer

### Step 6: 拒绝记账（Failed Edit Buffer）

被验证集门控拒绝的编辑提案，移入 `10-rejected-edits-buffer.md`。

**buffer 记录格式**：
```markdown
## Reject #[序号] — YYYY-MM-DD

### 提案
[原始编辑提案全文]

### 拒绝原因
- 验证集 S_old: XX% → S_new: XX%
- 具体失败分析：哪条规则导致分数下降
- 是否相同问题在历史中已出现：[是/否 + 历史条目号]

### 后续建议
- 这条规则是否可以缩小范围后再试？
- 是否有更精细的条件可以避免误伤？
```

### Step 7: Slow/Meta Update（周/月级别）

**慢更新（slow update）**：
- 每周日（周复盘 cron: 9af717d3c89b）执行
- 用整周轨迹 vs 上周 skill 版本做对比
- 将过去7天的验证结果分成四类：
  - ✅ **改进了** — 哪些编辑确实提升了命中率
  - ❌ **退化了** — 哪些编辑反而降低了某些场景的命中率
  - ⏳ **持续失败** — 哪些问题经历多轮仍未解决（需要新思路）
  - ✓ **稳定成功** — 哪些规则已经连续多天验证有效（可固化）

**元规则（meta rules）**：
- 写入 `03-review-experience.md` 末尾的「长期经验」章节
- 只记录稳定的、跨联赛的通用规则
- 不同于日常编辑（可能被下一轮拒绝覆盖），元规则不会随日常更新被冲走
- 元规则的生效需要 **连续 3 个验证周期**（日复盘验证）通过

## 二、编辑预算控制（Text Learning Rate）

### 预算表

| 周期 | 最大编辑数 | 说明 |
|:----:|:---------:|:-----|
| 每日复盘 | 3 条 | 只修最显著的偏差 |
| 每周复盘 | 5 条 | 可做跨联赛规则迁移 |
| 月度复盘 | 8 条 | 可做结构性的方法论调整 |

### 预算衰减

同一条规则连续被拒绝 2 次以上 → 该规则自动进入「冷却周期」（2周内不再考虑同类提案）。

**cool-down 记录也在 `10-rejected-edits-buffer.md` 中维护。**

## 三、拒绝记账复用机制

每次提出新编辑提案前，先查询 `10-rejected-edits-buffer.md`。

**禁止重试的情况**：
- 过去 2 周内已有相同提案被拒绝 → 跳过
- 相同规则在不同场次被拒绝 2 次 → 将范围缩小后再提案

**可以重试的情况**：
- 有新的轨迹数据出现，且数据量足以覆盖被拒绝时的顾虑
- 提案范围缩小到特定联赛/条件（而非泛化规则）
- 基础模型/执行环境有变化

## 六、使用示例

### 日复盘完整循环（以 2026-05-29 挪超6场为例）

**Step 1**: Rollout 已完成（5月29日预测输出）
**Step 2**: 轨迹采集 — 6场挪超实际结果
**Step 3**: Minibatch — 挪超共6场，分组分析发现：
  - 5/6场触发4b → 仅1场走预期方向
  - 4/6场平局（67%，远超~19%赛季均值）
**Step 4**: 编辑提案（3条）：
  1. replace: 因子4b在挪超从P0降为P1+λ修正
  2. add: 因子16「挪超平局化」规则
  3. add: 因子17「挪超让球-1穿盘区分」
**Step 5**: 验证门控：
  - S_old (无这些规则): 方向命中率 3/6 = 50%
  - S_new (有这些规则): 方向命中率 5/6 = 83%（模拟回测）
  - ✅ 通过 → 写入 v4.6.0→v4.6.1
**Step 6**: 驳回了第4条提案「挪超弱队主场λ×0.85」→ 进入 rejected buffer
