# 赛后回溯数据采集与欧指信号交叉验证方法

> 2026-06-18 从6/17芬超5场复盘沉淀
> 适用场景：竞足/世界杯赛后回溯预测验证

---

## 核心原则

复盘预测不能仅依赖预测文件做对比。**必须实时拉取赛后数据，交叉验证欧指信号的真实性**，才能区分「预测判断错误」和「预测时数据采集错误」。

## 数据流

```
okooo开奖页 ──→ 赛果（比分/半场/SP）
      ↓
匹配预测场次 ←── 预测文件（`~/.hermes/predictions/jingzu/YYYY-MM-DD.md`）
      ↓
新浪API欧指 ──→ 赛前采集的指数变化数据
      ↓                   ↓
新浪API欧指 ──→ 赛后实时再次查询同一matchID
      ↓
比对「预测时的指数变化」vs「赛后实时指数变化」
      ↓
判定：✅一致 → 预测数据可信，分析判断偏差
     ⚠️大变 → 预测数据本身有问题，需修正分析基准
```

## 步骤详解

### 第1步：从 okooko 获取赛果

```bash
curl -s "https://vxbf.okooo.com/kaijiang/sport.php?LotteryType=SportteryScore&LotteryNo=YYYY-MM-DD" \
  -H 'User-Agent: Mozilla/5.0 (Linux; Android 14; Pixel 8) AppleWebKit/537.36' \
  -H 'Referer: https://www.okooo.com/' | iconv -f GBK -t UTF-8
```

解析时注意：okooo按销售日期（不是比赛日期）分组。周三比赛可能归入周二tab。

**稳定解析方法（Python正则，2026-06-18验证通过）：**
```python
import re, sys
html = sys.stdin.read()
rows = re.findall(r'<tr>(.*?)</tr>', html, re.DOTALL)
for row in rows[1:]:  # 跳过表头行
    tds = re.findall(r'<td[^>]*>(.*?)</td>', row, re.DOTALL)
    if len(tds) < 6: continue
    num = re.sub(r'<[^>]+>', '', tds[0]).strip()
    home = re.sub(r'<[^>]+>', '', tds[1]).strip()
    away = re.sub(r'<[^>]+>', '', tds[3]).strip()
    score_match = re.search(r'<b[^>]*>([\d:]+)</b>', tds[4])
    score_full = score_match.group(1) if score_match else ''
    half_match = re.search(r'半场([\d:]+)', tds[4])
    score_half = half_match.group(1) if half_match else ''
    sp_match = re.search(r'<span[^>]*>([\d.]+)</span>', tds[5])
    sp = sp_match.group(1) if sp_match else ''
    print(f'{num} | {home} vs {away} | {score_full} | 半{score_half} | SP{sp}')
```

### 第2步：从新浪API获取赛前SPF → 匹配matchID

```bash
curl -s "https://mix.lottery.sina.com.cn/gateway/index/entry?format=json\
&__caller__=wap&__version__=1.0.0&__verno__=10000\
&cat1=jczqMatches&gameTypes=spf&date=YYYY-MM-DD&isAll=1&dpc=1"
```

响应结构：`d['result']['data']` → 列表，每项含 `matchId`, `spf`（赛前SPF逗号分隔）, `rqspf`（让球SPF）.

### 第2.5步：通过SPF值匹配场次与matchID（关键步骤）

新浪API返回的matchID（如3635511）与预测文件中的场次号（如3202）**完全不同**，必须通过SPF值的相似度来匹配。

**匹配方法**：
```python
# 从预测文件读取每场SPF
pred_spf = {
    3201: (2.25, 3.10, 2.76),
    3202: (1.46, 4.10, 5.00),  # → 匹配新浪3635511
    3203: (4.65, 3.60, 1.57),
    3204: (2.25, 3.20, 2.69),
    3205: (2.18, 3.26, 2.75),
}

# 从新浪API获取每场的SPF
# 响应 matchId=3635511, spf="1.46,4.10,5.00" → 匹配3202 ✓
# 响应 matchId=3635514, spf="2.25,3.21,2.68" → 匹配3204 ✓
```

**辅助匹配字段**：
- **让球数（rqspf）**：`RQ=-1`或`RQ=+1`是强区分符（如TPS受让=+1，其他让球=-1）
- **竞彩官方主胜**：新浪欧指数据中首家公司为"竞彩官方"，其o1New值（主胜赔）与SPF主胜接近
- **联赛名**：新浪API `footballMatchDetail` 返回`leagueName`可二次确认

**实战经验**：SPF完全匹配的是精确匹配；SPF不完全匹配时结合让球数字段判断（如TPSvs库奥皮奥预测RQ=+1，新浪RQ=+1确认）

### 第3步：对每个matchID查询欧指数据

```bash
curl -s "https://mix.lottery.sina.com.cn/gateway/index/entry?format=json\
&__caller__=wap&__version__=1.0.0&__verno__=10000\
&cat1=footballMatchOddsEuro&matchId=$MATCHID"
```

响应结构：`d['result']['data']` → 列表，每项含：
- `companyName` 公司名
- `o1Ini`/`o1New` 主胜初赔/即赔
- `o2Ini`/`o2New` 平局初赔/即赔
- `o3Ini`/`o3New` 客胜初赔/即赔

**关键计算**：
```python
# 百家平均
avg_1 = sum(float(o['o1New']) for o in data) / len(data)

# 指数变化 - 升/降家数统计
up1 = sum(1 for o in data if float(o['o1New']) > float(o['o1Ini']))
down1 = sum(1 for o in data if float(o['o1New']) < float(o['o1Ini']))
```

### 第4步：比对预测数据 vs 实时查询数据

将第3步实时查询的指数变化数据，与预测文件中记录的指数变化数据逐项对比：

| 维度 | 预测时数据 | 赛后实时数据 | 判定 |
|:----|:---------|:-----------:|:----:|
| 百家平均 | 2.487/3.331/2.635 | 2.456/3.335/2.659 | ✅ <5%偏差 |
| 胜指数变化 | 41升/3降 | 38升/3降 | ✅ 模式一致 |
| 负指数变化 | 6升/35降 | 8升/34降 | ✅ 模式一致 |

### 第5步：一致性判定标准

| 差异程度 | 判定 | 行动 |
|:--------|:-----|:-----|
| 百家平均偏差<5% 且 指数变化模式一致 | ✅ **预测数据可信** | 正常分析偏差原因 |
| 百家平均偏差<5% 但 指数变化略有强弱差异 | ⚠️ **临场漂移** | 标注「信号漂移」，仍可分析方向性 |
| 百家平均偏差≥5% 或 指数变化方向相反 | ❌ **数据不一致** | 标记「预测数据存疑」，需检查采集时间/路由 |
| 百家平均偏差显著 且 指数变化完全相反 | 🔴 **数据错误** | 跳过该场次精度统计，单独标注 |

### 第6步：逐场回溯预测精度

对数据可信的场次，做五维对比：

```python
常见回溯对比模板（v4.16.0 — 精简版）：

| 维度 | 预测值 | 实际值 | 判定 |
|:----|:-------|:------:|:----:|
| 正常比分 | X-X / Y-Y | N-N | ✅/❌/[偏差N球] |
| 单选方向 | 胜/平/负 | 胜/平/负 | ✅/❌ |
| 双选方向 | A/B | 实际 | ✅/❌ |
| 让球 | 让X | 让X(含推导) | ✅/❌ |
```

### 赛后欧指数据的时间窗口注意事项

1. **比赛结束后的欧指数据会重置**：一些博彩公司的赔率会在赛后固定为"1.00"或"已结算"，不再显示赛前即赔。因此赛后查询的指数变化数据可能比赛前少几家公司。
2. **指数变化在赛后可能被抹平**：部分数据源在赛后会移除临时赔率变动记录，只保留初赔和终赔。这可能导致赛后查询的"升/降家数"与赛前实时采集的有所差异。
3. **最佳验证窗口**：比赛结束后2~4小时内查询欧指数据，此时赔付已结算但历史变动记录尚未被清理。
4. **替代验证**：如果欧指指数变化数据不可用，使用百家平均作为替代验证指标——百家平均在赛后仍保持稳定。

### 数据流（2条并行线）

```
线A: 赛果采集

  → 对匹配到的matchID查footballMatchOddsEuro → 获取百家平均+指数变化

```

### Step 2：跨源matchID桥接

通过SPF值+让球数双重确认关联新浪API的matchID与预测场次：

```python
# 精准匹配：SPF主胜值完全一致（如1.46=1.46）
# 模糊匹配：偏差<5% + 让球数一致（如RQ=-1）
# 实战案例：
#   3202 SPF 1.46/4.10/5.00 ↔ 新浪3635511 SPF 1.46/4.10/5.00 ✅
#   3204 SPF 2.25/3.20/2.69 ↔ 新浪3635514 SPF 2.25/3.21/2.68 ✅
```

让球数字段（RQ）是强区分符：`+1`（主队受让）vs `-1`（主队让球）。

### Step 3：λ精算验证（交叉乘积反查）

对每场回溯，用实际赛前数据重新计算λ，验证预测的λ值合理性：

```python
# 交叉乘积公式
# λ₁ = (主队攻击λ/1.35) × (客队防守λ/1.35) × 1.35 × 因子修正

# 案例：3202 坦山猫 5-0 雅罗
# 坦山猫主场场均3.0球 → 攻击因子=3.0/1.35=2.22
# 雅罗垫底防守场均失2.5 → 防守因子=2.5/1.35=1.85
# λ₁ = 2.22 × 1.85 × 1.35/1.35 × 1.15(因子20) = 4.72
# Poisson(4.72): 4-0(19.4%)/5-0(18.3%) → 精确覆盖5-0 ✅

# 案例：203 雅罗 2-5 赫尔辛基（尾事件）
# 赫尔辛基客场攻击≈2.6球/场 → 因子=2.6/1.35=1.93
# 雅罗主场防守≈2.4球/场 → 因子=2.4/1.35=1.78
# λ₂ = 1.93 × 1.78 × 1.35/1.35 = 3.44 (或更激进4.62)
# λ₁ = 雅罗主场攻击≈0.6球/场 → 因子=0.6/1.35=0.44
# P(雅罗2球) with λ₁=0.60 = 8.8% → **低概率尾事件**
```

### Step 4：极端赛果分类（尾事件标记）

当实际比分不在泊松Top2双选内时，计算概率P值分类：

| P值范围 | 分类 | 判定 |
|:--------|:-----|:-----|
| P ≤ 1% | 🔴 极低概率事件 | 模型无法预期，标记尾事件 |
| 1% < P ≤ 5% | 🟠 低概率事件 | 偶发偏差，不计入系统性偏差 |
| 5% < P ≤ 10% | 🟡 边缘事件 | 检查是否有遗漏因子 |
| P > 10% | ✅ 正常范围 | 模型应覆盖，改进优化 |

**公式**：P(组合) = Poisson_P(主队进球, λ₁) × Poisson_P(客队进球, λ₂)

**实战**：203 雅罗2-5赫尔辛基 — λ₁=0.60, λ₂=4.62 → P(2,5)=8.8%×19.9%=1.75% → 🟠尾事件
