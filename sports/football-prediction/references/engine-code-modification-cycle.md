# 引擎代码修改循环方法论（v1.0）

> 2026-07-05 从v10.24→v10.27的6轮迭代中总结。适用场景：当复盘发现偏差后，需要从分析→代码→验证的完整闭环。

## 核心循环

```
复盘偏差 → 第一性原理根因分析 → 引擎代码修改 → 全量回测验证 → 回归检查
```

## 六步工作流

### Step 1: 拉实时数据（零缓存）

禁止使用之前轮次的缓存数据。每次修改前重新拉取新浪API三接口：

1. `footballMatchOddsEuro?matchId=X` — 53家初赔vs即赔 → 8参数(r1/c1/r2/c2/r3/c3)
2. `footballMatchDetail?matchId=X` — 排名/轮次/天气/中立场
3. `footballMatchOddsAsia?matchId=X` — 17家亚盘+大小球

### Step 2: 第一性原理根因分析

从原始信号出发，不问"预测为什么错"，而是问"市场信号告诉我们什么，引擎为什么没正确响应"：

| 模式 | 问题 | 分析方法 |
|:-----|:-----|:---------|
| **信号缺失** | 引擎没处理某种信号模式 | 检查`apply_all_corrections()`中是否有匹配的条件分支 |
| **信号冲突** | A信号和B信号矛盾 | 检查执行顺序：晚执行的覆盖早执行的 |
| **修正系数不足** | 方向正确但比分偏差大 | 检查λ修正系数(×1.08~×1.20)，与实际情况的gap |
| **噪声信号** | 在不适用的场景中触发 | 检查豁免条件(o1<1.30超深盘等) |

**⚠️ 2026-07-05关键发现：对称信号遗漏**
原引擎代码只处理了`c1(买主)`和`r3(卖客)`两个方向的λ修正，但遗漏了它们的对称信号：
- `ad/c3(买客)` → 应λ_a×factor（对称于c1买主）
- `xh/r1(卖主)` → 应λ_a×factor（对称于r3卖客）
- 检查方法：手动列出8参数所有6个方向(r1/c1/r2/c2/r3/c3)，对照代码看看哪些被处理了哪些被漏了

### Step 3: 引擎代码修改

修改位置：`scripts/league-predict.py` 中的 `apply_all_corrections()` 函数

修改原则：
1. **不破坏已有正确预测** — 增加新分支而非修改旧逻辑
2. **新因子放在末尾** — 后执行的修正覆盖先执行的
3. **对称修正放在F30/F31/F33后面** — 避免与专用反转规则冲突
4. **赛后评分器(predict_match中)兜底** — 当λ修正不够时，通过排序调整Top2

**执行顺序铁律：**
```
F30(三极端) → F31(骑墙) → F33(芬超ad) → 对称信号(ad买客/xh卖主)
```
如果对称信号在F30之前执行，会与F30的反转方向冲突（如ad=45买客在F30恢复主队前错误拉高λ_a）。

### Step 5: 偏差收敛判断

基线先行，修改后全量回测，逐场对比。收敛到方向100%+精确60%+后，剩余偏差需判断系统偏差vs泊松残差。

### Step 6: 数学极限判断

当λ修正系数已到×1.15~×1.20后仍偏差，且泊松概率<8%时，属于>2σ正常残差，不应继续调整（防过拟合）。

## 本循环案例

| 轮次 | 精确 | 方向 | 关键发现 |
|:----|:---:|:----:|:---------|
| 基线v10.24 | 36% | 73% | — |
